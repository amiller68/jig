## Goal

Extract the Linear API client from `crates/jig-core/src/issues/linear_client.rs` (1,139 lines) into a clean `linear` module under `crates/jig-core/src/linear/`. The current file mixes GraphQL query strings, response deserialization structs, request building, and client methods all in one file.

The new module should be an opinionated GraphQL client for Linear with:

1. **`linear/mod.rs`** — re-exports, `LinearClient` struct (holds API key, base URL, ureq agent)
2. **`linear/error.rs`** — `LinearError` enum (own error type, NOT `crate::error::Error` — see pattern in `git/error.rs` and `host/tmux.rs`)
3. **Request-per-file pattern** — each GraphQL operation gets its own file describing:
   - The GraphQL query/mutation string
   - The request input types (if any)
   - The response deserialization structs
   - A method on `LinearClient` that executes it

For example:
- `linear/queries/list_issues.rs` — query string, filter inputs, response types, `LinearClient::list_issues()`
- `linear/queries/get_viewer.rs` — viewer query + response
- `linear/mutations/create_issue.rs` — mutation string, input struct, response, `LinearClient::create_issue()`
- etc.

4. **`LinearClient` struct** should have a builder or simple constructor taking API key, and a private `fn execute<T: DeserializeOwned>(&self, query: &str, variables: Value) -> Result<T>` that all operations delegate to.

## Current operations to extract (11 total)

Queries: ListIssues, GetIssue (reuses list), Viewer, ListWorkflowStates, TeamByKey, LabelsByTeam, ProjectsByName

Mutations: UpdateIssueStatus, UpdateIssue, CreateIssue, IssueRelationCreate, IssueRelationDelete

## Constraints

- The `issues/linear_provider.rs` (LinearProvider, implements IssueProvider trait) stays in `issues/` — it wraps the new `linear::LinearClient`. Update its imports.
- Follow the module error pattern: `LinearError` enum in `linear/error.rs` with `type Result<T> = std::result::Result<T, LinearError>`. Add `#[from] LinearError` to `crate::error::Error`.
- The identifier parsing utils (`parse_identifier`, team key extraction) can live in `linear/mod.rs` or a `linear/identifier.rs`.
- Must pass `cargo build`, `cargo clippy`, `cargo test`.
- Do NOT add new dependencies.
